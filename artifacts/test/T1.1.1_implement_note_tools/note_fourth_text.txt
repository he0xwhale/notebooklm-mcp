🌟 数字化审计实操建模全景解析：虚列工程项目（完整推演版）第一部分：业务通识篇（看透“虚列工程”的本质）1. 核心业务痛点：账本里的“障眼法”正常的基建工程必然包含两大部分开销：“买东西的钱”（物料成本）和**“买服务的钱”**（施工费、设计费等）。如果出现极端失衡，通常暗示着巨大的合规风险：物料占比极低（<5%）：钱全花在无形的服务上。这往往是在“套取资金”，或者把日常运维费用违规塞进了基建项目。物料占比极高（>95%）：几乎全是买材料的钱。这可能是为了年底突击花钱而疯狂囤积物资，或者工程根本是伪造的。2. 审计制度红线这种乱立项、虚列成本的行为严重违反相关管理办法，审计必须通过数据计算，揪出这些涉嫌虚列、套取资金的异常项目。

第三部分：积木式 SQL 建模与数据沙盘推演我们将上述 6 条原始数据喂给经过优化的模块化 SQL，观察它如何在每一步进行“数据提纯与判决”。-- ==========================================
-- 🎯 步骤 1：圈定目标项目库（大浪淘沙）
-- 动作：利用 DISTINCT 将成百上千的流水账去重，并截取 pspid 前两位锁定主/配网项目，生成唯一的项目台账主表。
/* 
   ▶ 【Step1 样例数据输出】(成功去重，3个项目各留1行)：
   | dw01_mc     | pspid  | xmmc        |
   |-------------|--------|-------------|
   | 国网XX公司  | 130001 | A市主网工程 |
   | 国网XX公司  | 150002 | B县配网工程 |
   | 国网XX公司  | 160003 | C镇造假工程 |
*/
-- ==========================================
WITH Step1_TargetProjects AS (
    SELECT DISTINCT 
        dw01_dm, dw01_mc, dw02_dm, dw02_mc, 
        dw03_dm, dw03_mc, dw04_dm, dw04_mc, 
        bukrs, butxt, pspid, xmmc
    FROM rds_xydszhsjpt.dwd_fin_gccbmxb
    WHERE substr(pspid, 1, 2) IN ('13','15','16','17','18','19','1R','C1','C5','C6')
),

-- ==========================================
-- 🎯 步骤 2：计算分子“物料总成本”（挑出买东西的钱）
-- 动作：只汇总物料编码 (matnr) 不为空的流水金额。
/* 
   ▶ 【Step2 样例数据输出】(注意：C工程完全没物料，在此表消失！)：
   | pspid  | sum_wl (物料成本) |
   |--------|-------------------|
   | 130001 | 3 万              |
   | 150002 | 40 万             |
*/
-- ==========================================
Step2_MaterialCost AS (
    SELECT pspid, SUM(wtgbtr) AS sum_wl
    FROM rds_xydszhsjpt.dwd_fin_gccbmxb
    WHERE matnr <> '' AND matnr <> ' ' AND matnr IS NOT NULL
    GROUP BY pspid
),

-- ==========================================
-- 🎯 步骤 3：计算分母“实际总成本”（所有的开销）
-- 动作：不对流水做任何过滤，直接按项目加总所有发生金额。
/* 
   ▶ 【Step3 样例数据输出】：
   | pspid  | sum_zcb (总成本)  |
   |--------|-------------------|
   | 130001 | 100 万            |
   | 150002 | 100 万            |
   | 160003 | 100 万            |
*/
-- ==========================================
Step3_TotalCost AS (
    SELECT pspid, SUM(wtgbtr) AS sum_zcb
    FROM rds_xydszhsjpt.dwd_fin_gccbmxb
    GROUP BY pspid
),

-- ==========================================
-- 🎯 步骤 4：数据大融合与占比预计算（打上终极补丁）
-- 动作：以Step1为主表，左关联Step2和3。使用 COALESCE 防御 NULL 陷阱，使用 CASE WHEN 防御除数为 0 报错。
/* 
   ▶ 【Step4 样例数据输出】(宽表拼接完成，算出占比)：
   | pspid  | xmmc        | 物料成本   | 总成本 | 物料成本占比 |
   |--------|-------------|------------|--------|--------------|
   | 130001 | A市主网工程 | 3 万       | 100 万 | 3.00         |
   | 150002 | B县配网工程 | 40 万      | 100 万 | 40.00        |
   | 160003 | C镇造假工程 | 0(补丁生效)| 100 万 | 0.00         |
*/
-- ==========================================
Step4_CalculateRatio AS (
    SELECT 
        t1.*,
        -- 【实战补丁1：COALESCE】如果左关联不到物料成本(如C工程)，强制转换为 0
        ROUND(COALESCE(t2.sum_wl, 0), 2)  AS 物料成本,
        ROUND(t3.sum_zcb, 2) AS 总成本,
        
        -- 【实战补丁2：CASE WHEN】总成本不为0才除，避免系统崩溃
        CASE 
            WHEN t3.sum_zcb <> 0 THEN ROUND(COALESCE(t2.sum_wl, 0) / t3.sum_zcb * 100, 2) 
            ELSE NULL 
        END AS 物料成本占比
    FROM Step1_TargetProjects t1
    LEFT JOIN Step2_MaterialCost t2 ON t1.pspid = t2.pspid
    LEFT JOIN Step3_TotalCost t3    ON t1.pspid = t3.pspid
)

-- ==========================================
-- 🎯 步骤 5：终极审判，落法槌（WHERE 门卫过滤）
-- 动作：仅放行占比 >95% 或 <5% 的极端数据。
/* 
   ▶ 【最终输出的审计疑点报告】(B工程 40% 被正常过滤，抓出 A 和 C)：
   | 项目编码 | 项目名称    | 物料成本 | 总成本 | 占比 | 判定结论 (人工分析辅助)             |
   |----------|-------------|----------|--------|------|-------------------------------------|
   | 130001   | A市主网工程 | 3 万     | 100 万 | 3.00 | <5%，涉嫌违规将服务费列入基建       |
   | 160003   | C镇造假工程 | 0 万     | 100 万 | 0.00 | <5%，极端无物料异常，高度涉嫌套现   |
*/
-- ==========================================
SELECT
    dw01_dm AS 一级单位代码, dw01_mc AS 一级单位名称,
    dw02_dm AS 二级单位代码, dw02_mc AS 二级单位名称,
    dw03_dm AS 三级单位代码, dw03_mc AS 三级单位名称,
    dw04_dm AS 四级单位代码, dw04_mc AS 四级单位名称,
    bukrs AS 公司代码, butxt AS 公司名称,
    pspid AS 项目编码, xmmc AS 项目名称,
    物料成本, 总成本, 物料成本占比
FROM Step4_CalculateRatio
WHERE 物料成本占比 > 95 OR 物料成本占比 < 5;

第四部分：核心技术与避坑指南（进阶思维总结）通过完整的数据推演，我们可以深刻总结出数字化审计实战中的 3 个高级思维：DISTINCT 名片去重器： 底层明细表是流水账，同一工程有成千上万条记录。在步骤 1 构建基础名单时如果不使用 DISTINCT 进行“降维”，会导致后续结果呈几何级数膨胀，制造大量垃圾数据并瘫痪数据库。分清 CASE 与 WHERE 的合作流水线：CASE WHEN 是“收银员”：不删除数据，只负责给每一行安全地算账，把总成本为 0 的异常废账变为安全的 NULL 标签。WHERE 是“门卫”：负责最后的拦截，遇到走极端的比率直接放行曝光，遇到 NULL 则因无法比较大小直接无视丢弃，双剑合璧过滤了“数据噪音”。致命的 NULL 陷阱（边界测试思维）： 如沙盘推演中的 C镇造假工程，纯造假套现项目根本没有物料记录。在使用 LEFT JOIN 时，它的物料成本会返回 NULL。如果不使用 COALESCE(t2.sum_wl, 0) 将其强制转换为数字 0，这个造假最严重的项目就会在数学计算中逃脱系统的追捕。
